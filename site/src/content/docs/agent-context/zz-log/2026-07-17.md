## Session: the 800-agent Sol burn — incident + guard

### What happened

A `fablexryu`-style session (first prompt: `ultracode`; main model Fable 5) on the mobile-development cyberchaste project spawned **~790 research subagents between 21:00–23:00 on 07-16** — deep-research-style prompts ("Research only, no file edits. Find public battery-impact promises…"), 818 agent transcripts / 237 MB total under the session dir. Every agent inherited the `CLAUDE_CODE_SUBAGENT_MODEL=gpt-5.6-sol` floor baked into fablex → **~50% of the Codex weekly quota**. The Fable orchestrator made 314 API calls managing it → **100% of the Claude 5h session**.

### Root cause chain (all four links required)

1. **`ultracode` standing opt-in** (the `u` launcher variants) — every session pre-authorizes multi-agent orchestration; no per-fan-out consent.
2. **Deep-research fan-out** — the harness happily scales to hundreds of finders; the built-in caps (16 concurrent / 1000 lifetime) are throughput rails, not budget rails.
3. **fablex Sol floor** — all of it routed to the Codex lane.
4. **Quota asymmetry** — Claude's bucket refills in ~5h; **Codex draws from a weekly pool**. Same mistake, 20× the blast radius. (This is the resurrection of the exact failure the 2026-06-13 tiering doctrine came from — "deep research gave it to a bunch of fable agents" — the floor fixed the *per-agent cost*, claudex/fablex silently re-broke the *aggregate* by pointing the floor at a weekly-metered provider.)

### Response (deterministic layer first, per firing-reliability doctrine)

- **`agent-guard` global PreToolUse hook** on `Agent|Workflow` — hard cap: 40 spawned subagents per session per rolling 5h window. Hooks fire even under `--dangerously-skip-permissions`, which is why this is a hook and not a permissions entry (all the y-launchers bypass permission prompts). Tested: allows normal sessions, blocked the runaway session's state (exit 2 with instructive stderr), 1h override flag works, historical agents age out of the window. Tunables: `CLAUDE_AGENT_CAP` env at launch; `agent-cap-off`/`agent-cap-on` bashrc toggles. Canonical: `profiles/claude-global/agent-guard.sh`; installed: `~/.claude/scripts/agent-guard.sh` + `~/.claude/settings.json` hooks entry.
- **CLAUDE.md Hard Rule: fan-out disclosure** — >15 expected subagents ⇒ state count/model/token estimate and get explicit confirmation in that turn; ultracode opt-in does not waive it.
- **Quota-asymmetry section** added to the claudex pattern doc — treat the Sol lane as weekly budget; mass fan-outs belong on the 5h-refill side or need explicit budget caps.

### Also this session

- **disk-slim bug found+fixed**: `bun pm cache rm` needs a package.json in cwd and silently no-oped → the 5 GB bun cache survived every past sweep (and explains a chunk of "the plague returns"). Now `rm -rf ~/.bun/install/cache` directly. Manual sweep freed ~5 GB in-guest; docker remains the whale (23 GB images / 0 active containers, 8.7 GB build cache — full `docker system prune -af --volumes` left as the user's deliberate call; VHDX gap ~22 GB pending a compact-wsl.bat run).
- **KB coherence audit** (the ultracode ask that started the day): 28-agent workflow (Sonnet, ~2.1M subagent tokens — pre-guard; would pass the new cap at 28<40) audited CLAUDE.md/AGENTS.md/skills/patterns/bashrc-comments against the claudex-arc developments. Confirmed findings preserved in `zz-research/2026-07-17-kb-audit-findings.md`; application deferred by the incident. Top confirmed: CLAUDE.md per-delegation bullet omits the resolution order and mis-frames `sol` as a universal target; AGENTS.md project-bootstrapper claims it "can invoke seacow-scaffolder" (agents can't spawn agents); delegation-advisor skill lacks any model-selection guidance.

## Session: ccx — interactive launcher picker (fzf front end over the cc*/claudex/fablex family)

The suffix-letter alias family (`ccrys`, `fablexry`, `claudexyu`, …) is write-only: composing a launch means mentally encoding brain × session × permissions × subagent-tier into letters. New `ccx` command walks the same four axes as fzf menus, with the quota-lane/billing consequence shown as a preview pane next to each option (Codex = weekly pool, fablex = needs both pools healthy — the 07-16/17 burn lessons ride in the picker text itself).

- New snippet: `profiles/bashrc-snippets/agent-launcher.sh`, sourced in `~/.bashrc` right after `claude-code-helpers.sh` (it composes the existing `claudex`/`fablex` functions — adds no new lane).
- Session axis includes a **pconv-backed picker**: lists this cwd's sessions with title/date/msg-count read straight from the `.jsonl`s (immune to the stale sessions-index bug; falls back to native `--resume` if pconv/jq absent). This is `ccrh`'s index-bypass idea upgraded from "blindly newest" to "choose with eyes".
- Subagent-tier menu only appears on the native-Claude lane (proxy lanes pin Sol); includes `manual` = the `ccrym` no-floor mode via `env -u`.
- `ccx -n` dry-runs (prints the composed command). Verified: syntax, heredoc tab integrity (fzf `\t` delimiters survive the `<<-` tab-strip), stubbed-fzf dry-run composes `CLAUDE_CODE_SUBAGENT_MODEL=sonnet claude`, and the jq row-builder returns real sessions for the obsidian vault cwd.
- Context that prompted it: a `claudex --resume <uuid>` today landed in a *fresh* session in the wrong cwd after Sol rejected the 548-msg resume with a context-window 400 — choosing sessions blind is exactly the failure mode the pconv picker axis removes.
- Follow-up ergonomics: `x` is the one-character alias (`xn` dry-run). Enter-mashing is now intentionally useful: native Claude → **continue most recent** → **skip permissions** → Sonnet agents. Both menus bind `j/k` and `Ctrl-j/Ctrl-k` to down/up; headers expose the controls. Verified parser acceptance and the full default composition: `CLAUDE_CODE_SUBAGENT_MODEL=sonnet claude --continue --dangerously-skip-permissions`.
- WSL slowness diagnosis: host C: is 95% full (50 GB free), guest is 60 GB used, compacted VHDX is 74 GB — only a 14 GB gap remains, so its size is now broadly honest rather than failed compaction. The live metadata micro-test measured ext4 `/tmp` at **0.01s** vs `/mnt/c` at **3.27s** for 1,000 tiny files (~327× slower); git status on the `/mnt/c` scaffold alone took 1.72s. Active development trees living on DrvFS are the dominant latency mechanism, compounded by the nearly-full host SSD. Guest usage is mostly two Docker daemons/caches: rootless data root 22 GB (23 GB images + 8.7 GB build cache accounting semantics overlap) plus root daemon 10.3 GB images/3.9 GB build cache/2.9 GB unused volumes; no containers active. Also 3.5 GB pip cache. Cleanup remains deliberate — diagnosed, not deleted.
