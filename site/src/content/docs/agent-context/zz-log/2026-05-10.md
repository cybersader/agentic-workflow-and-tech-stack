## Session: GitHub-Actions-via-Tailscale → Dokploy webhook (the pattern)

The `cybersader/openclast` docs site is on this Dokploy (the tailnet-only one). Wired up autodeploy from `git push` → live. The naive "Dokploy GitHub App + autodeploy toggle" flow does NOT work when Dokploy is tailnet-only — GitHub fires webhooks from public IPs, can't reach `dokploy.<your-tailnet>.ts.net`, deliveries time out silently. The workaround pattern that does work, end to end, is documented here for the next app I deploy.

### The shape

```
git push (public)
    ↓ webhook → public GitHub Actions runner
    ↓ tailscale/github-action@v3 (joins ephemeral runner to tailnet, tag:ci)
    ↓ curl POST → http://dokploy.<your-tailnet>.ts.net/api/deploy/<token>
    ↓ Dokploy validates branch + watchPaths
    ↓ deploy
```

### Two-layer auth

- **Network layer** (Tailscale OAuth + tag ACLs): the runner is ephemeral but joins as `tag:ci`. ACL grants `tag:ci → tag:dokploy:80,443,3000`. If the runner is compromised, blast radius is limited to whatever the ACL exposes.
- **App layer** (Dokploy refresh-token in URL): the per-app deploy URL has a long random token in the path. Both layers must pass — neither alone is sufficient.

### Concrete recipe

**Tailscale admin** (one-time, ~5 min):

1. `https://login.tailscale.com/admin/settings/oauth` → Generate OAuth client. Scope: `auth_keys` write. Tags: `tag:ci`. Copy ID + Secret (secret shown once).
2. ACL — using the **grants** format (Tailscale's newer unified syntax — older `acls` block doesn't exist on this tailnet):
   ```jsonc
   "tagOwners": {
     "tag:ci":      ["autogroup:admin"],
     "tag:dokploy": ["autogroup:admin"]
   },
   "grants": [
     {
       "src": ["tag:ci"],
       "dst": ["tag:dokploy"],
       "ip":  ["tcp:80", "tcp:443", "tcp:3000"]
     }
   ]
   ```
   **Gotcha:** in the grants format, each `ip` entry holds ONE port or range — `"tcp:80,443,3000"` is rejected with `only one port range allowed`. Split into separate strings in the array.
3. Machines tab → `dokploy` → Edit ACL tags → add `tag:dokploy`.

**Dokploy** (per app):

4. Get the webhook URL from the app — Dokploy shows it as `http://dokploy:3000/api/deploy/<token>` (the internal Docker service hostname). Hand-edit the hostname when storing it as a GitHub secret: `http://dokploy.<your-tailnet>.ts.net/api/deploy/<token>`.
5. **Setting `Settings → Web Server → Server Domain` does NOT template the webhook URL** in v0.29.x — it stays as `dokploy:3000`. Hand-edit is the only path. (Setting Server Domain to `dokploy.<your-tailnet>.ts.net` was harmless — it adds a Traefik route on port 80 for the admin UI but doesn't affect the webhook URL output. HTTPS toggle must stay OFF — Let's Encrypt can't reach a tailnet-only host.)

**GitHub repo** (per repo):

6. Three repository secrets — exact names matter:
   ```
   TS_OAUTH_CLIENT_ID
   TS_OAUTH_SECRET
   DOKPLOY_DEPLOY_URL    ← the hand-edited URL
   ```
7. Workflow file:
   ```yaml
   name: Deploy to Dokploy
   on:
     push:
       branches: [main]
       paths: ['<deploy-path>/**', '.github/workflows/deploy-*.yml']
     workflow_dispatch:
   jobs:
     deploy:
       runs-on: ubuntu-latest
       timeout-minutes: 5
       steps:
         - uses: tailscale/github-action@v3
           with:
             oauth-client-id: ${{ secrets.TS_OAUTH_CLIENT_ID }}
             oauth-secret:    ${{ secrets.TS_OAUTH_SECRET }}
             tags:            tag:ci
         - env:
             SHA: ${{ github.sha }}
           run: |
             curl -fsSL --max-time 30 -X POST \
               -H 'Content-Type: application/json' \
               -H 'X-GitHub-Event: push' \
               -d "{
                 \"ref\": \"refs/heads/main\",
                 \"head_commit\": {\"id\": \"${SHA}\", \"message\": \"CI deploy\"},
                 \"commits\": [{\"id\": \"${SHA}\", \"modified\": [\"<deploy-path>/.deploy-trigger\"]}]
               }" \
               "${{ secrets.DOKPLOY_DEPLOY_URL }}"
   ```

### Dokploy-payload quirks (the non-obvious part)

Reading Dokploy's source (`apps/dokploy/pages/api/deploy/[refreshToken].ts` on `Dokploy/dokploy@1c6fdc1`) was required to reverse-engineer the right payload. Dokploy's deploy webhook is *not* a "generic trigger" endpoint — it expects to look like a real provider webhook event:

- **`extractBranchName(headers, body)`** returns null unless one of `x-github-event`, `x-gitea-event`, `x-gitlab-event`, `x-event-key` (Bitbucket), or `x-softserve-event` is present in headers. Without one, the branch check fails with `{"message":"Branch Not Match"}` even when the body is a perfect GitHub-shaped JSON.
- For `sourceType: "github"` apps, the branch is extracted as `body.ref.replace("refs/heads/", "")` and compared against `application.branch`. Must match exactly.
- For `sourceType: "git"` apps (generic Git URL, not provider-linked), it's compared against `application.customGitBranch`.
- **`shouldDeploy(watchPaths, normalizedCommits)`** runs upstream of branch matching for some sourceTypes. `normalizedCommits = body.commits.flatMap(c => c.modified)`. If the app has watch paths set, our payload's modified files must satisfy them. Workaround: include a sentinel path under the watched dir (`docs-site/.deploy-trigger` worked).

### Failure modes I hit, in order

| Symptom | Cause | Fix |
|---|---|---|
| `Error: OAuth identity empty` | Repo secret name typo / saved as Variable not Secret | Names must be exact: `TS_OAUTH_CLIENT_ID`, `TS_OAUTH_SECRET`. GitHub UI has separate Variables/Secrets tabs — must be Secrets. |
| `{"message":"Branch Not Match"}` (empty POST) | No body | Add `-d '{"ref":"refs/heads/main"}'` |
| `{"message":"Branch Not Match"}` (with body) | Missing provider header — `extractBranchName` returns null | Add `-H 'X-GitHub-Event: push'` |
| Curl times out at TCP layer | ACL not granting `tag:ci` → `tag:dokploy` | Verify dokploy host has `tag:dokploy` applied; verify grants block exists |
| Dokploy returned `Watch Paths Not Match` | App has watchPaths configured but payload's modified[] is empty | Add a sentinel path under the watched dir to `commits[0].modified` |

### Why it mattered

This is the unblocker for "private repo + tailnet Dokploy" as a deploy topology. Without it, every push needs a manual button-click in the Dokploy UI — viable for one repo, not viable as the project count grows. With this pattern wired up once per repo (~10 min of admin clicks + a 25-line workflow file), every subsequent project on this Dokploy gets webhook-driven autodeploy that respects the tailnet boundary.

The pattern itself is sanctioned by Tailscale — they ship `tailscale/github-action` specifically for this, and "deploy to private infra from public CI" is the canonical example in their docs. It only feels like duct-tape because Dokploy doesn't surface it as a first-class option in its UI; once wired, it's solid.

## Session: Testing-discipline anti-pattern — "env-fragile" as a copout

While shipping v0.1.6 Phase 3 of the Crosswalker plugin (custom Bases view), I claimed the WebdriverIO E2E suite was "env-fragile, deferred" after my first `bun run e2e` timed out at 5 minutes. That conclusion was wrong:

- The first run was **downloading the Obsidian binary** (~150 MB)
- The Crosswalker project's `testing-patterns` SKILL.md explicitly noted "first run downloads ~150 MB; subsequent runs use cache"
- I didn't read carefully enough; I didn't retry with a longer timeout; I didn't inspect the cache state before drawing the conclusion

**The correction**: re-tried with a 240s per-spec timeout. Smoke passed in 58s. Full 17-spec suite passed in 18:04. Backfilled 13 real E2E tests for Phases 2 + 3 of v0.1.6 (post-shipment) — the most meaningful one being `idempotent first-run write preserves user edits across plugin disable/re-enable cycle`, which exercises an architectural-commitment property (Settled #3 user-edit-safety) that no unit test with a mock vault could verify.

### The anti-pattern (generalized — applies far beyond Obsidian plugins)

When a project has multiple configured test surfaces (Jest unit + WebdriverIO E2E + Playwright + obsidian-cli, in the Crosswalker case), it's tempting to declare a phase "done" after running just the fast surfaces. That's how regressions ship.

The discipline:

- **Read the project's `testing-patterns` skill in full BEFORE claiming any surface is unavailable.** First-run download costs, env requirements, and known gotchas usually live there.
- **If a surface "appears fragile," diagnose why** before concluding it can't be used. Common causes mistaken for "env-fragile": first-run binary downloads, missing display server, missing dependencies, fixtures not yet built. Most are fixable in minutes.
- **"I ran X but it timed out" is not the same as "X doesn't work"** — the timeout might be the wrong signal. Inspect what was happening when it timed out.
- **Per-phase**: every milestone in a project's roadmap should ship at LEAST one E2E spec proving the feature works in the target runtime. This is what catches the regressions unit tests miss.

### What I'm pulling into the scaffold's `testing-patterns` skill

Added a new anti-pattern section to `01-kernel/skills/testing-patterns/SKILL.md` ("Declaring a Test Surface 'Env-Fragile' Without Diagnostic Effort") with the diagnostic checklist (cache state, display server, first-run download, missing deps) so future agents on any project don't repeat the mistake.

### Why this lands here (the meta point)

The Crosswalker project shipped because the user pushed back: *"Did you test as much as you could without me having to manually do it. We should have WebDriver and also the Obsidian CLI for testing."* That single push corrected what would otherwise have shipped as latent gap. The agentic workflow scaffold's job is to make that push unnecessary — encoding the discipline at the kernel level so the agent never claims a surface is fragile without trying.

The lesson stays with the agentic kernel; the project-specific tests live in the Crosswalker repo. Bidirectional mention: this log entry references the Crosswalker delivery; the Crosswalker zz-log entries note the E2E backfill.

## Session: Drop Omnisearch HTTP for Obsidian CLI search (mid-session pivot)

After shipping Sessions 1–3 of today's work, two real-world facts forced a refactor:

1. **Omnisearch HTTP is unreachable from WSL by default.** `curl http://localhost:51361` and `curl http://<windows-gateway>:51361` both time out — Omnisearch binds to `127.0.0.1` on the Windows host running Obsidian, and WSL's network namespace doesn't see that. Fixable with `.wslconfig` mirrored-networking-mode changes, but that's a system-wide change for a single feature.
2. **Most of cybersader's "registered project trees" aren't Obsidian vaults.** Of the 37 paths registered with ck, roughly half are vanilla code/data trees with no `.obsidian/`. An Obsidian-only search layer simply doesn't apply to them — they need a vault-agnostic tool.

User flagged the second point explicitly: *"And some of these aren't obsidian vaults So our tooling would have to work without having to use obsidian in most places."* That reframes the design from "two parallel tracks" to "ck is universal, Obsidian CLI is an accelerator for the vault subset."

### The replacement: Obsidian CLI native `search` / `search:context`

Probed `/mnt/c/Users/<you>/AppData/Local/Obsidian/Obsidian.com` via WSL→Windows interop. Obsidian 1.12+'s CLI exposes `search` and `search:context` commands with `format=json` and `vault=<name>` args. From the test:

- `obs-status` → "Obsidian CLI is available... vaults reachable: 30"
- `obs-search "wikilink" 3` → `[".claude/skills/workflow-scaffold/SKILL.md", ".claude/skills/testing-patterns/SKILL.md", ".claude/skills/seacow-conventions/SKILL.md"]`
- `obs-search-context "firing reliability" 2` → JSON with `{file, matches: [{line, text}]}` — line numbers and the exact matched lines, across `skill-patterns/SKILL.md`, `proactive-patterns/SKILL.md`, `local-search/SKILL.md`.

This is *strictly better* than the Omnisearch HTTP shape:
- Works from WSL without any config
- Returns line context (Omnisearch HTTP doesn't)
- No plugin toggle, no port conflicts
- Multi-vault aware via `vault=<name>` (30 vaults reachable)
- Native to Obsidian — no plugin dependency

Downside: only available for paths that are Obsidian-registered vaults. Which is fine — that's exactly the "accelerator for the vault subset" the user described.

- `profiles/bashrc-snippets/local-search-helpers.sh` — refactored. Track A (Omnisearch HTTP) replaced with `_obs_cli` resolver + `obs-search` / `obs-search-context` / `obs-vaults` / `obs-status` using Obsidian CLI. Dropped `OMNISEARCH_PORT` env var. The `cks-*` family (Track B) unchanged — it was already vault-agnostic.

## Session: cks fan-out output cleanup

Smoke-testing `cks` (regex fan-out across all 37 registered paths) revealed three usability issues:

1. **No tree attribution.** Output showed `./CLAUDE.md` lines — could be any of 37 projects.
2. **Errors silenced.** `2>/dev/null` swallowed everything; couldn't tell if a tree failed.
3. **No first-query announcement.** `cks-sem` would silently start an index build, leaving the user wondering why the query was slow.

Fixed in one pass:

- Refactored `cks` / `cks-sem` / `cks-hybrid` to share an `_cks_run_one` helper that captures stdout/stderr separately via temp files, prints a `── <tree-name> ── (/abs/path)` header before each tree's matches, rewrites `./relative` paths in ck's output to absolute (using `sed`), suppresses output for empty trees (no headers for trees with zero matches), and routes real errors to stderr without aborting the fan-out.
- Added `_cks_announce_index` helper that prints `(building index — first query for this tree, may take a moment…)` to stderr when an un-indexed tree is about to be queried via `cks-sem` / `cks-hybrid`.
- Tied off the "No matches found" stderr noise — ck prints that on empty results with exit 1; the helper now greps for it and treats it as silent success (only true errors get flagged with ⚠).
- Stream-redirection idiom in the initial refactor (`err=$( { out=$(...); } 2>&1 1>&3 3>&-)`) silently broke output capture — switched to plain tempfile capture. Simpler and correct.

Verified end-to-end:

- `cks "wikilink"` → cleanly grouped output across the trees that actually have matches; absolute paths throughout; quiet on trees with zero matches.
- `cks "AGENTS" -l` → file lists per tree, attributed cleanly to source trees.
- `_cks_run_one <indexed-tree> "--sem" "skill firing reliability" --topk 2` → semantic results, properly attributed.

Updated `02-stack/01-ai-coding/local-search-ck-and-obsidian-cli.md` with an "Output shape from the fan-out helpers" section. Updated `.claude/skills/local-search/SKILL.md`'s Output shape entry to match.

### Why this mattered enough to break out into its own session

This was the actual test-driven part. Sessions 1–4 designed and shipped the helpers; Session 5 made them *usable*. Tree attribution + silent-on-empty + first-query announcement together turn `cks-sem` from "I have no idea what it just did" to "I can see exactly where each match came from and why a query was slow."

## Session: Tailscale file-share — extract four-flavor recipe from a pconv'd session

Dogfooding the new local-search stack: `pconv list` surfaced a recent conversation at `…/4 VAULTS/b&g_vault/BACKUP/b&g/30 - CYBERSADER` that contained a clean four-flavor reference for tailnet-based file sharing. Hadn't been canonized into the scaffold; promoting it.

Shipped:

  - `tdrop <file> <peer>` — Taildrop send

  - `tshare <dir> [port]` — tailnet-only HTTPS via `tailscale serve`; spawns python3 http.server, records PID

  - `tshare-status` — `tailscale status` + serve + funnel + local http.server pidfiles, one screen
  - `TAILSCALE_CLI` env var lets you point at `tailscale.exe` (Windows-side from WSL) instead of `/usr/bin/tailscale` if you want files to land on the Windows host's node identity.
