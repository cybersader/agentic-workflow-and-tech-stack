## Session: WSL/Docker VHDX bloat — 50 GB reclaimed, pattern promoted

C: hit 25.5 GB free (2.7%). WizTree treemap (fetched from the Zipline host via Windows-side PowerShell — WSL's resolver can't see `.home` domains, itself a reusable gotcha) showed `ext4.vhdx` at 71 GB and `docker_data.vhdx` at 28.6 GB, plus large non-VHDX offenders.

**Diagnosis:** inside WSL, `df` reported 53 GB used vs the 71 GB VHDX — ~18 GB phantom allocation from the VHDX one-way ratchet. On top: two Rust `target/` dirs on /mnt/c (22 GB + 9.5 GB, regenerable), a **second docker daemon inside Ubuntu** (`~/.local/share/docker`, 9.7 GB, 20 images / 0 containers — separate from Docker Desktop's VHDX), `~/.claude/debug` at 3.4 GB (single files up to 577 MB), and ~6 GB of npm/bun/playwright/apt caches.

**Reclaimed live (this session):** both `target/` dirs (~31.5 GB back on C: immediately — /mnt/c frees instantly, no VHDX involved), docker prune (9.7 GB → 83 MB), debug-log trim (3.4 GB → 76 KB), caches cleared. In-guest usage 53 → 37 GB. `fstrim` + `wsl --shutdown` + `--set-sparse true` + docker_data.vhdx compaction handed to the user as a PowerShell finale (running it from inside WSL would kill this very session — the recipe documents that footgun).

**Promoted straight to tier-2** (evidence gate: verified execution with receipts) as `02-stack/patterns/wsl-vhdx-bloat-reclaim.md` — the ratchet mechanics, the ordered recipe, the double-daemon gotcha, prevention (sparse mode, build trees out of /mnt/c, log rotation), and the honest Linux-comparison verdict: the VHDX ritual is WSL-specific; about half the bloat was platform-independent hygiene debt.

Adds to the accumulating WSL-liability ledger (tailscale-serve NAT asymmetry, localhost binding, 9P slowness, now the VHDX ratchet) relevant to any future native-Linux migration decision.

## Session 2: recurrence-proofing — disk-doctor / disk-slim / SessionStart warning

User: the bloat "comes back like the plague." Reframed from incident to treadmill: normal churn re-inflates everything, and a documented ritual is the weakest prevention layer. Shipped the automation per the firing-reliability doctrine (hook = trigger, helper = content):

- **`disk-doctor`** (bashrc) — read-only diagnosis: host free, guest usage, ext4.vhdx-size-vs-guest-used gap (the compaction bounty), BOTH docker daemons, usual-suspect caches, cargo `target/` hunt on /mnt/c.
- **`disk-slim`** (bashrc) — the whole 2026-07-11 sweep as one command (debug-log trim >7d, conservative docker prune, npm/bun/playwright/apt caches, fstrim), with freed-MB report. Deliberately does NOT auto-delete cargo targets or run the aggressive prune.
- **`check-disk.sh` SessionStart hook** (machine-local settings.local.json) — every new session warns when C: < 60 GB free or guest > 70%, pointing at disk-doctor. Tested live (fired correctly at 44 GB). The plague now gets caught at ~15% free, not 2.7%.
- Pattern doc gained a "When it keeps coming back" section documenting the three layers and what automation deliberately leaves manual.

Verified post-reclaim state: C: 25.5 → 44 GB free before the user even runs the VHDX finale (~34 GB more waiting in compaction + the 28.6 GB docker_data.vhdx).

## Session 3: "sparse came back OFF" — the flag is not sticky, so verify it

User reported having enabled sparse VHDX before, yet the plague returned. Checked empirically: `fsutil sparse queryflag` on the live ext4.vhdx → **NOT sparse**. Mystery solved — the flag doesn't persist across distro re-registrations (fresh `{GUID}` VHDX), can be reset by WSL updates/moves, and the old experimental `.wslconfig` `sparseVhd=true` only ever applied to newly created VHDXs (pulled upstream after corruption reports). Also confirmed the guest mounts with `discard`, so once sparse is genuinely ON, TRIM propagates automatically.

Systemic fix: **verify, never assume** — `disk-doctor` now checks the sparse flag via fsutil on every run and prints ON/OFF with the exact fix command (tested live: correctly reports ✗ OFF right now). Pattern doc's recurrence section updated with the non-stickiness evidence and causes. The SessionStart low-disk hook remains the backstop for when any single layer silently fails — which is exactly what happened here.

## Session 4: the finale script — wsl-reclaim.ps1

The host-side steps require `wsl --shutdown`, which kills the agent session — so the finale must run unattended by the agent. Shipped `tools/wsl-compact/wsl-reclaim.ps1`: an interactive, admin-required PowerShell script that does the whole sequence with per-step prompts — shutdown → sparse on Ubuntu **with fsutil verification that the flag actually took** (the not-sticky lesson from Session 3) → enumerate every *.vhdx across WSL/Docker/podman roots with sizes → diskpart-compact the chosen ones → optional podman-machine unregister → optional NVIDIA DXCache clear → before/after C: free report. Ends by telling the user to run `disk-doctor` in the restarted WSL to confirm `sparse: ✓ ON`.

Authoring gotcha worth remembering: writing PowerShell through a bash heredoc mangled the escaping (backtick-quote sequences) and produced parse errors — rewrote byte-exact via python and validated with the real PowerShell AST parser (`[Parser]::ParseFile`) before handing it over. "Parses clean" is the minimum bar for a script the agent can't supervise.

## Session 5: sparse VHD is disabled upstream — recommendation inverted

Post-finale disk-doctor showed the win (C: 44 → 74 GB free; ext4.vhdx 72 → 49 GB via compaction) but `sparse: OFF` *again*, despite the .bat running set-sparse. Captured the actual refusal by running the command live: **"Sparse VHD support is currently disabled due to potential data corruption"** — WSL 2.7.x requires `--allow-unsafe` to force it. This retroactively explains the user's entire recurrence history: every sparse attempt ever made silently failed (the .bat swallowed the refusal text).

Recommendation inverted accordingly: **compaction cadence over sparse.** Forcing a flag named `--allow-unsafe` on the primary dev distro is the wrong trade for this stack's values. Updated: disk-doctor (OFF now reported as *expected* with the cadence pointer, not an alarm), compact-wsl.bat (step 3 now surfaces the choice explicitly instead of failing silently — defaults to skip), wsl-reclaim.ps1 (same, with --allow-unsafe only behind an explicit informed prompt), pattern doc (sparse section rewritten with the refusal text; prevention section leads with the cadence).

Final tally for the whole incident: **C: 25.5 → 74 GB free** (+48.5 GB) via in-guest sweep + /mnt/c artifact cleanup + VHDX compaction, with the recurrence now handled by hook-warned cadence instead of a fix that never actually worked.

## Session 6: full automation — scheduled auto-compact with safety interlock

User: "it would be nice if it's actually fully automated." The constraint: compaction requires `wsl --shutdown`, so the automation must live host-side and must never kill live work. Shipped two scripts to `tools/wsl-compact/`:

- **`compact-wsl-auto.ps1`** — unattended, double-gated: (1) threshold gate — exits unless C: free < 80 GB (no pointless churn); (2) **safety interlock** — exits if interactive processes (claude/zellij/tmux/ssh) are running inside Ubuntu. Only then: fstrim → shutdown → diskpart-compact ext4.vhdx (+ docker_data.vhdx when Docker Desktop isn't running) → append to `%LOCALAPPDATA%\wsl-compact.log`. `-DryRun` evaluates the gates without acting; admin enforced at runtime only for the real path so dry-runs work unelevated.
- **`install-scheduled-task.ps1`** — registers "WSL VHDX Auto-Compact", Sundays 04:00, elevated (S4U, no stored password). Weekly firing is safe because the script self-gates — most weeks it logs "skip".

**Verified live:** both parse clean (PS AST parser); dry-run correctly *deferred because it detected this very Claude session* (interlock count=9), including when forced past the threshold with `-MinFreeGB 999`. The failure mode that matters — killing active work — is the one demonstrably guarded.

Remaining manual step (one-time, needs elevation the agent doesn't have): run the installer from an admin PowerShell. After that the loop is closed: SessionStart hook = in-your-face warning; Sunday task = silent maintenance; disk-doctor = on-demand truth; compact-wsl.bat = manual full-service with backup.
