## Session: fullscreen TUI renderer baked into the ai-coding stack doc

Discovered (via a terminal-emulator research thread) that Claude Code v2.1.110 shipped an alt-screen + per-frame-diff TUI renderer — `/tui fullscreen` (or the older `CLAUDE_CODE_NO_FLICKER=1` env var). It replaces the default inline renderer that repaints the whole region every frame (the flicker source, worst on narrow panes / over multiplexers) and anchors the input box at the bottom. Tested live in this session — markedly better for long agent sessions in Zellij.

`/tui fullscreen` self-persists: the command writes the `tui` setting to config and relaunches, so it survives restarts without an env var.

Shipped:

- `02-stack/01-ai-coding/index.md` — new "TUI rendering — turn on fullscreen (no-flicker)" subsection under the Claude Code config cascade. Documents the enable/revert commands, the env-var-vs-setting conflict (prefer the `/tui` setting, unset `CLAUDE_CODE_NO_FLICKER`), the alt-screen scrollback/copy tradeoff, the reported CJK-on-copy corruption caveat, and the recommendation to default it on for this stack.
- `agent-context/zz-research/2026-04-23-claude-code-ink-narrow-pane-bug.md` — added an "Update 2026-06-13" section. The fullscreen renderer fixes *flicker* definitively, but whether it also fixes the narrow-pane mid-word-wrap *corruption* documented in that note is untested (different failure mode, same rendering layer). Logged a retest action: next narrow-pane (phone-reattach) session, retest under `/tui fullscreen`; if fixed, the note moves observed → resolved. Bidirectional cross-link added.

**Why tier-2:** the fullscreen recommendation is general Claude Code config wisdom, no personal data. Ships to the public mirror via the `02-stack/` allowlist.

**Why it mattered:** it's a strictly-better default that was opt-in (alt-screen takes over scrollback/copy, so Anthropic shipped it off-by-default). Worth baking the recommendation into the stack so the next environment rebuild turns it on without rediscovering it. The "always run it like this" instinct is right — it's now the documented default for the stack.

**Follow-up (same session):** made it automatic on rebuild, not just documented. Added step "2b. Set the TUI renderer to fullscreen" to `03-work/rebuild/04-ai-tools.md` — a jq-merge (or create-file) snippet that writes `"tui": "fullscreen"` into `~/.claude/settings.json` deterministically during the AI-tools rebuild step, so a fresh environment gets it before first launch without anyone running `/tui fullscreen` by hand. Confirmed the live global settings.json on this machine already carries `"tui": "fullscreen"` (written when the command was run earlier), so the key/value is verified correct, not guessed.

## Session: subagent model tiering — stop premium main sessions from fanning out premium subagents

**The pain:** user ran a "deep research" delegation from a Fable 5 main session; it fanned out to many subagents that inherited the premium tier and burned a ton of context + cost. Confirmed the mechanism by reading Claude Code docs (via a Haiku guide agent — cheap, and a live demo of the principle) and web sources: subagents default to `inherit` (the main model); inheritance is capped at Opus, so a Fable main spawns Opus-tier subagents; and 2026 Dynamic Workflows can fan out to tens–hundreds of them. Many premium agents × full context windows each = the blowup.

**The fix (two layers, both shipped):**

1. **`CLAUDE_CODE_SUBAGENT_MODEL=sonnet` env var** in `~/.bashrc` — the global floor. This is the real fix for the incident because it catches the **built-in** `Explore` / `Plan` / `general-purpose` agents (the "deep research" delegation was one of these — no frontmatter to pin). Documented resolution order: env var > per-invocation `model` > agent frontmatter > main model. Override per-shell with `unset`.
2. **Per-agent frontmatter retune** — all custom agents were pinned to `opus` (still premium). Changed every one `opus → sonnet` across the project `.claude/agents/` (9 files, committed) and the live global `~/.claude/agents/` (13 files, machine-local). Kept `improvement-logger` on `haiku` (pure capture). Net: no subagent, custom or built-in, runs on `opus`/`fable` by default anymore. Two files (`agent-builder`, `skill-builder`) had CRLF line endings that the first sed missed — fixed.

**Docs:**
- `.claude/skills/agent-patterns/SKILL.md` — new "Model Tiering — never let a subagent inherit a premium main model" section: the trap, the resolution order, the two-layer fix, the `haiku`-for-grunt / `sonnet`-for-reasoning / never-`opus`/`fable`-for-subagents rule of thumb. Canonical shape: Fable/Opus orchestrator → Sonnet implementation → Haiku scan/extraction.
- `03-work/rebuild/04-ai-tools.md` — new step 2c sets `CLAUDE_CODE_SUBAGENT_MODEL=sonnet` on a fresh rebuild so the floor is automatic, not rediscovered.

**Why it mattered:** the default (subagents inherit the premium main model) is a cost/context trap that only bites when a task fans out — exactly when you least want it. The principle "premium tier is for the orchestrator; delegate grunt work cheap" is now baked into the scaffold's shipped agent frontmatter AND the rebuild flow, so a fresh environment gets it without the surprise-bill discovery. Bonus: the diagnostic itself used a Haiku agent, dogfooding the principle.

**Follow-up (shorthands):** added launch-time subagent-tier aliases to `profiles/bashrc-snippets/claude-code-helpers.sh`, extending the existing `cc`/`r`/`y` scheme with a trailing tier letter — `ccrys` (Sonnet subagents), `ccryo` (Opus), `ccryh` (Haiku), all built on the `ccry` workhorse (continue + skip-perms). Plus a flexible `ccm <tier> [args]` function. A launch-time `CLAUDE_CODE_SUBAGENT_MODEL=` prefix beats the `~/.bashrc` floor for that one session, so `ccryo` opts *up* to Opus subagents when a project warrants it. Complements the per-project `.claude/settings.json` `env` block (automatic-per-directory) vs these (choose-at-launch).
