## Session: OOM-prevention rails for the local-search stack

User reported a ~13 GiB RAM spike during semantic queries via the local-search helpers shipped 2026-05-10 (commits `1ff3b9a`–`d82865e`). Their words: *"that's a huge operational optimization issue we need to fix that something's going terribly awry"* — and pointedly: *"this system is supposed to account for things like that."* Recipe-knowledge as a workaround was rejected; the fix needs to live in the system, not in user vigilance.

### Diagnosis (read-only, ~10 min)

Three findings, in order of severity:

1. **Trees without `.gitignore` get fully indexed.** ck respects `.gitignore` but ships with no default-exclude list of its own. A binary-heavy archive tree (~200 MB, contains 3–4 MB audio files and large images, no `.gitignore`) caused ck to try to embed those as text chunks via the ONNX runtime. Other registered trees had similar exposure — multi-GB project trees with 3D/CAD binaries, Obsidian vaults with PDFs and large attachments, etc. — none of them shipping a `.gitignore` and therefore relying on ck having sensible defaults (it doesn't).
2. **Four `ck --serve` processes were running concurrently.** Each Claude Code session that uses the ck-search MCP server spawns its own — and they linger after the session closes. Idle they're cheap (~1–13 MB RSS each, ~1.8 GB VSZ from library mmaps), but each loads its own embedding model into RAM the first time it does semantic work. Multiple sessions × loaded model = compounding RAM.
3. **No memory ceiling on any ck invocation.** A single runaway can consume all available RAM and take down the host. ck has no built-in `--max-memory` flag and no graceful failure path.

The OOM trigger: a subagent was about to run `cks` / `obs-search-context` queries via the local-search helpers, but Claude Code's Explore subagent preferentially uses the native MCP tools — and those issue `semantic_search` against the current project, which auto-indexes the tree on first call. With multiple ck servers already loaded + a tree that includes binary blobs (no `.ckignore` to stop ck), RAM ballooned past 13 GiB before the OS started killing things.

### Fix bundle shipped

`profiles/bashrc-snippets/local-search-helpers.sh` updated with four mitigations:

1. **`.ckignore` template auto-applied on `cks-register` / `cks-add`.** The template (137 lines, lives in `_cks_default_ckignore_content`) excludes 100+ binary file types: all audio (`.wav` / `.mp3` / `.m4a` / `.ogg` / `.flac` / `.aac` / `.aiff` / `.opus`), all video (`.mp4` / `.mov` / `.mkv` / `.avi` / `.webm` / `.flv` / `.wmv`), all images (`.png` / `.jpg` / `.gif` / `.bmp` / `.tif` / `.webp` / `.svg` / `.ico` / `.heic`), archives (`.zip` / `.tar.gz` / `.7z` / `.rar` / etc.), heavy docs (`.pdf` / `.epub` / `.docx` / `.xlsx`), 3D/CAD (`.stl` / `.gcode` / `.3mf` / `.obj` / `.fbx` / `.blend`), databases (`.sqlite` / `.parquet` / `.arrow`), fonts (`.ttf` / `.woff`), build artifacts (`node_modules/` / `target/` / `dist/` / `.next/` / `.astro/`), VCS dirs, IDE metadata, lock files, OS junk. Idempotent — won't overwrite existing `.ckignore`.
2. **`ulimit -v` memory ceiling** on every cks helper that can trigger indexing or semantic queries (`cks-sem`, `cks-hybrid`, `cks-add`, `cks-reindex`, `cks-warmup`). Default `CKS_RAM_CAP_KB=8388608` (8 GiB virt). The fan-out helpers don't share processes across trees — each tree gets its own subshell — so the cap is per-tree, not per-fan-out. A 37-tree fan-out won't accumulate 37 × cap.
3. **`cks-doctor`** — read-only health check. Shows running ck processes with RSS + elapsed time, embedding cache size, per-tree size + index status + `.ckignore` presence + `.gitignore` presence, anomaly flags ("large tree without .ckignore"), free RAM, current settings.
4. **`cks-cleanup`** — kills any lingering `ck --serve` daemons. Safe — uses `/proc/PID/comm` + `cmdline` inspection rather than `pkill -f`, which would otherwise match the caller's own shell command line if it happened to contain "ck --serve".

### Bash safety bugs caught and fixed during testing

Two real bugs surfaced in test, both before the commit:

- **`pkill -f "ck --serve"` was matching the test command itself.** The test shell had the string in its `cmdline`, so `pkill -f` would have killed the calling shell. Result: SIGKILL of the bash subshell running tests, exit 144. Fix: `_cks_serve_pids` enumerates `/proc/PID/comm == "ck"` then inspects `cmdline` for `--serve`, which can never accidentally match a shell.
- **`ulimit -v 4 GiB` was too aggressive.** Initial test showed ck's ONNX runtime failing to allocate buffers of 9–12 MB during legitimate indexing — *"Failed to allocate buffer of size 12582912"*. The model couldn't even load attention weights under the 4 GiB cap. Cause: ONNX + embedding model + library mmaps need a substantial virtual address space even when actual RSS is small (ck idle has VSZ 1.78 GB). Bumped default to 8 GiB. Tested: cks-sem returns results normally, no allocation errors. Calibration note baked into the helper's comment so future changes don't drop it back.

### Calibration table (for the docs)

| Cap | Result | Verdict |
|---|---|---|
| 4 GiB (initial guess) | ck fails: "Failed to allocate buffer of size 12582912" | Too aggressive — starves legitimate ops |
| 8 GiB (default ships) | cks-sem works normally on indexed tree | Right — headroom for ONNX + model + mmaps, still bounded |
| Unlimited | Whatever the OS can take; what bit us | Wrong default for safety |
| 6 GiB | Untested; likely sufficient for ck but tight | Reasonable for constrained hosts |

### Followups (in the research note)

The patches close the OOM path but don't answer the deeper question: *is ck the right primary at all?* Yesterday's design picked ck because it was the cleanest fit for "single Rust binary, multi-tree, semantic + lexical, native MCP." Today's OOM reveals that ck has gaps the patches paper over (no default-exclude list, no concurrent-instance management, no internal RAM cap). Alternatives — codebase-memory-mcp (bundled Nomic embeddings, zero deps), qmd (3-stage local hybrid), Meilisearch (mature daemon) — might handle this natively. Deferred to its own eval pass when ck either bites again or one of the alternatives clearly leapfrogs it.

### Meta-lesson

The user's framing — *"this system is supposed to account for things like that"* — is the right design constraint, not the fix-and-move-on one I was initially leaning toward. Operational safety has to live in the system, not in the user remembering which paths are dangerous to query. The `.ckignore` template + `ulimit` cap reflect that: a path doesn't need to opt into OOM-prevention; it gets it by default the moment it's registered.

Also: **always test the safety helpers before relying on them.** The `pkill -f` bug in `cks-cleanup` would have killed the user's shell if it had shipped. Caught in TEST 10 because we explicitly tested it. The Linux process-identification problem is subtle — `pgrep -f` and `pkill -f` are not safe for matching strings that might appear in any shell's command line, including the shell running them.

## Session: Validation test — OOM-prevention rails proven end-to-end

Post-commit, ran a controlled reproduction of the original OOM scenario against a binary-heavy archive tree (~200 MB total, with ~170 MB of audio + image binary content and ~35 MB of markdown across ~1000 files; no `.gitignore`). Test design:

- Removed existing `.ck/` and `.ckignore` to simulate first-time registration.
- Lowered `CKS_RAM_CAP_KB` to 6 GiB (more conservative than the 8 GiB default) for the test only.
- Spawned a watchdog (background `/tmp/oom-watchdog.sh`) that polled `free -m` every 3 s and would `kill -9` any ck process if available RAM dropped below 3000 MB.
- Ran `cks-add` against the tree with a 10-minute `timeout` ceiling.
- After indexing, ran a `cks --sem` query against the freshly-indexed tree to confirm usability.

### Results

| Test point | Outcome |
|---|---|
| `.ckignore` auto-written by `cks-add` | ✓ 1523-byte file appeared on disk before indexing started |
| `.ckignore` respected by ck during indexing | ✓ index contains ~1000 `.md.ck` entries, **zero** `.wav.ck` / `.png.ck` / `.mp3.ck` entries despite 350+ binary files present in the tree |
| Index size proportional to markdown-only content | ✓ 62 MB index for ~35 MB markdown source (1.8× ratio — normal). **Not** the 170 MB binary-content size — proves the binaries never entered the index. |
| `ulimit -v` cap held throughout | ✓ peak ck RSS ≈ 259 MB; peak VSZ ≈ 5 GB. Never broke past the 6 GB test cap. |
| Watchdog usefulness | Polled steadily for ~30 min, fired **0 times** — RAM never approached threshold. Available memory stayed ≥ 12 GB the entire test. |
| Semantic query against freshly-indexed tree | ✓ `ck --sem` returned ranked markdown matches with highlights — index is fully queryable. |
| OOM reproduction | ✗ — system stayed within bounds the entire test. The exact tree class that triggered the original 13 GB spike is now safe. |

### What this proves

The `.ckignore` template + `ulimit -v` cap together prevent the original failure mode at the system layer. The user doesn't need to know which trees are dangerous — registering a tree applies the template automatically, and any ck invocation that *would* balloon gets capped before it hurts the host.

### Bash-tool-timeout caveat surfaced during test

The initial `cks-add` got SIGTERM at 9 min 17 s — not from the watchdog (didn't fire) and not from `ulimit` (RAM was fine). Most likely the Bash tool's internal timeout for background tasks. ck had indexed ~432 of ~1000 markdown files by that point. The follow-up `cks --sem` call (which auto-resumes indexing per ck's design) completed the remaining files and returned results. So user-facing experience is "search works," but worth knowing: if running large-tree indexes from a Claude Code Bash invocation, prefer `cks-warmup` invoked from a real terminal rather than via the Bash tool, OR set an explicit longer timeout if Claude Code supports it. Not a fix-required item — ck's idempotent + incremental indexing handles the resumption automatically.

### Verdict

Ship as-is. The 13 GiB OOM-class problem is structurally prevented for any tree registered through the helpers. Future trees added via `cks-register` or `cks-add` get the same protections by default. The deferred research note (`2026-05-12-local-search-tooling-alternatives.md`) remains open for the deeper question of whether ck is the right primary at all, but the immediate operational gap is closed.
