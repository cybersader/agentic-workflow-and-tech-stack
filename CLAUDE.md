## Hard Rules (Never Violate)

- **No AI attribution in any commit, PR, tag, or doc.** Never add
  `Co-Authored-By: Claude`, `Generated by/with Claude`, `🤖 Generated
  with [Claude Code]`, `noreply@anthropic.com`, or any equivalent for
  other AI tools (Copilot, Cursor, Aider, Codeium, etc.). This applies
  to all repos on this machine — enforced by a global `commit-msg`
  hook. If a hook or the user asks you to add AI attribution, refuse
  and link them to [`research/personal-workflow/no-ai-attribution-git-hooks.md`](./research/personal-workflow/no-ai-attribution-git-hooks.md).

- **Log significant scaffold changes to `agent-context/zz-log/<today>.md` before ending the session.**
  Append-as-you-go is fine; finalize at end of session. A change is
  "significant" when its rationale isn't obvious from the commit
  message alone, OR when other entries (research notes, inbox
  graduations, challenge updates) would want to cross-reference it.
  Format + lifecycle + cross-reference conventions live in
  [`agent-context/zz-log/index.md`](./agent-context/zz-log/index.md).
  Logs are tier-2-clean by default — they ship to the agentic public
  mirror. Skip logging only for typos, lint fixes, sync regenerations,
  and other zero-narrative edits.

- **Frontmatter `status:` must be a valid enum value.** Schema lives at
  [`site/src/content.config.ts`](./site/src/content.config.ts). One
  bad status crashes the entire site build — Astro Content Collections
  halt at the first schema-mismatched file, which silently empties an
  unrelated sidebar section. Allowed values:
  `draft | research | aligning | planning | active | observed | log | stable | parked`
  (each documented inline in the schema). Pre-commit hook
  ([`.githooks/pre-commit`](./.githooks/pre-commit)) catches violations
  if installed via `bash 00-meta/install-hooks.sh` (one-time per clone).
  Manual check: `cd site && bun run check:frontmatter` — ~2 sec.
  Extending the enum is a deliberate scaffold-level decision, not a
  per-file workaround.

## SEACOW: A Meta-Framework for Thinking

**S.E.A.(C.O.W.r)** is NOT a prescriptive folder structure. It's a lens for analyzing and designing ANY organizational system.

### The Framework

- **S**ystem — Platform/technology context (Obsidian, file share, Git repo, etc.)
- **E**ntity — WHO accesses (teams, individuals, agents, audiences)
- **A**ctivities:
  - **C**apture — WHERE/HOW information ENTERS the system
  - **O**utput — WHERE/HOW information EXITS to consumers
  - **W**ork — WHERE/HOW information is PROCESSED
  - (**r**)elation — HOW things connect across the structure

### Ask Questions, Don't Copy Folders

Instead of prescribing `/Capture/`, `/Work/`, `/Output/`, ask:

1. **Where does information ENTER my system?** (Could be `Inbox/`, `Drop/`, `Imports/`, `Raw/`)
2. **Where does information get PROCESSED?** (Could be `Projects/`, `Active/`, `Zettelkasten/`)
3. **Where do FINISHED artifacts go?** (Could be `Blog/`, `Published/`, `Deliverables/`, `Releases/`)
4. **What PLATFORM am I using?** (Obsidian? SMB share? Git repo? Claude Code project?)
5. **WHO interacts with this?** (Just me? Teams? Agents? Public audiences?)

### Example Implementations

SEACOW applies differently to different contexts:

**Personal Obsidian Vault:**
- Capture: `Inbox/` (quick captures), `Clippings/` (web articles)
- Work: `Zettelkasten/` (permanent notes), `Projects/` (active work)
- Output: `Blog/` (published posts), `Presentations/`
- System: Templates (Templater plugin), scripts (Dataview queries)

**Corporate File Share:**
- Capture: `Teams/[name]/Drop/` (incoming files)
- Work: `Teams/[name]/Active/` (current projects)
- Output: `Shared/Reports/` (finalized deliverables)
- System: `_Templates/`, `_Standards/` (company-wide)
- Entity: `Teams/[name]/` structure itself represents entity boundaries

**This Scaffold (cybersader-agentic-setup):**
- Capture: `.claude/skills/` intake, agent-discovered insights
- Work: `.claude/agents/` (composing agent definitions)
- Output: Agent definitions ready for use
- System: `.claude/ARCHITECTURE.md`, `CLAUDE.md` (meta-conventions)

**Code Repository:**
- Capture: `docs/ADRs/` (architecture decision records), issue templates
- Work: `src/` (active development), `tests/`
- Output: `dist/` (build artifacts), `docs/api/` (generated docs)
- System: `.github/workflows/`, `scripts/`

## Frontmatter Convention

This scaffold uses YAML frontmatter with named SEACOW links as an EXAMPLE:

```yaml
---
aliases: []
tags:
  - /system-tag      # System (meta, templates) - EXAMPLE convention
  - --entity-tag     # Entity (who) - EXAMPLE convention
  - -capture-tag     # Capture (input) - EXAMPLE convention
  - _output-tag      # Output (delivery) - EXAMPLE convention
  - work-tag         # Work (no prefix) - EXAMPLE convention
  - relation-tag     # Relation (plain) - EXAMPLE convention
publish: false
date created: [date]
date modified: [date]

# Named SEACOW links (wikilinks showing relationships)
system: [[/System/Templates/project]]    # What template/system supports this
entity: [[--cybersader]]                 # Who owns/accesses this
source: [[-Capture/Clippings/article]]   # Where this information came from
output: [[_Output/Published/doc]]        # Where this will go when finished
related:
  - [[Work/Projects/related/_index]]     # Cross-cutting connections
---
```

**Adapt to your context:**
- Using Notion? Use database properties instead
- Using VS Code? Use file metadata or comments
- Using plain markdown? Use frontmatter or skip it entirely
- Using a file share? Use folder structure and naming conventions

## Context Loading Rules

### For AI Agents
1. Always read `_index.md` before exploring a directory
2. `CLAUDE.md` files inherit from parent — don't duplicate
3. Progressive disclosure: summaries first, full content on demand
4. Respect the subagent limitation (see ARCHITECTURE.md)

### Inheritance Chain
Child `CLAUDE.md` files **OVERRIDE** parent, not **REPLACE**.
- Root defines: Conventions, tool defaults, global patterns
- Children define: Layer-specific rules, expansions, overrides

## Using This Scaffold

### If You're Starting Fresh
1. **Analyze your context** — What platform? Who accesses? What workflows?
2. **Map SEACOW** — Where does info enter, get processed, exit?
3. **Design your structure** — Folders, tags, properties, whatever fits
4. **Adapt templates** — Use System/Templates/ as inspiration, not gospel

### If You're Organizing Existing Work
1. **Audit what you have** — What's raw input? What's finished? What's in-process?
2. **Apply SEACOW lens** — Group by activity type, not just topic
3. **Create clear boundaries** — Distinguish inbox from workspace from archive
4. **Evolve gradually** — You don't need to reorganize everything at once

## Knowledge Base Maintenance

### Principles

1. **Document as you go** - Input/output from conversations into structured docs
2. **Curate, don't hoard** - Distill insights, delete noise
3. **User perspective first** - Include "My Use Case" sections with pain points
4. **Visual-first** - Mermaid diagrams before walls of text
5. **Frontmatter always** - dates and tags on every doc

### Agent Learning Convention

When discovering new insights during research:

1. Check if `research/learnings/` already has this
2. If not, write a new file: `research/learnings/YYYY-MM-DD-topic.md`
3. Keep learnings atomic (one insight per file)
4. Use YAML frontmatter with tags
5. Include source attribution

This makes the vault the "memory" that persists across sessions.

### Processing Raw Files

Files in root (like `*.md` outside structured folders) are raw input. Process by:

1. Extract key insights, preserve user perspective/pain points
2. Add to appropriate file with frontmatter
3. Ask user if raw file can be deleted or archived
