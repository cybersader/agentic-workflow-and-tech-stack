## Session: skill-firing-reliability principle baked into the scaffold

A separate project surfaced the canonical recursive-failure-mode for Claude Code skills: an agent shipped a buggy cache-row design because it skipped a referenced spec doc (`[[Ch 18 §2.5]]`); its proposed fix was to write a `wikilink-crawl` *skill* — i.e., to put the trigger for "must-crawl-wikilinks" inside the same probabilistic mechanism that just failed. Skills fire only when description-match wins on that turn, which is exactly the condition that had already failed. Putting a fix into a skill reproduces the original miss.

Decision encoded into the scaffold so the next person (or session) doesn't relearn it the hard way.

- `.claude/skills/proactive-patterns/SKILL.md` — added new top-of-file section **"Firing Reliability — Pick the Right Mechanism FIRST"** before "The Four Trigger Types." Contains: reliability ranking (hook → CLAUDE.md → slash command → subagent → skill, most → least), explicit decision rule keyed on must-fire vs should-fire, the wikilink-crawl example as the canonical failure mode, and the layering pattern that works (**CLAUDE.md or hook = trigger; skill = content**). Hybrid pattern noted: a `UserPromptSubmit` hook can inject a "consider invoking X skill" reminder to bridge reliability and content separately.
- `.claude/skills/skill-patterns/SKILL.md` — added a callout in the Purpose section pointing readers to the proactive-patterns reliability section before they write a new skill. Common failure mode named explicitly: "writing a skill to fix a bug caused by a skill not firing." Same layering pattern surfaced: trigger ≠ content.
- Project memory updated: `feedback_skill_firing_reliability.md` captures this as feedback-type guidance so future sessions apply it whenever skill-design comes up; MEMORY.md index updated.

**Why it mattered:** the user has explicitly said they've had a bad time with skills firing — independent of this scaffold work. The scaffold's `proactive-patterns` skill previously documented all four trigger types as if they were equivalent choices, with no reliability ranking. That undersold hooks and CLAUDE.md and oversold skills, which is exactly the misframe that produces the recursive-failure-mode bug. The new section makes the trade-off explicit before any trigger choice gets made.

**Where this lands tier-wise:** scaffold-level (tier-2-clean) — this is general Claude Code design wisdom, not cybersader-specific infrastructure. Ships to the agentic public mirror per the existing 02-stack-and-skills allowlist.

## Notes / observations

- The principle was generalized from a single in-the-wild failure plus the user's stated frustration. Worth re-examining after a few more weeks of usage to see whether the ranking holds or needs nuance (e.g., does a `Use PROACTIVELY when X` description actually fire materially more often than a passive description? — anecdote suggests yes; not yet measured).
- The `wikilink-crawl` example is canonical because it's the cleanest recursive-failure-mode case: the *fix* and the *bug* have identical mechanisms. Future scaffold edits should keep that example unless a clearer one surfaces.

## Session 2: agent + skill pairing pattern shipped as tier-2 doc

Following directly from Session 1 — the firing-reliability ranking surfaced an obvious next move: document the deterministic pattern that *exists in the workspace already* but isn't explicitly named anywhere. Agents preload skills via the `skills:` frontmatter field (8 of the workspace's meta-agents already do this), which collapses two probabilistic decisions into one routing decision. Skill content goes from "loaded if Claude judges it relevant" to "loaded every time this agent runs."

- `02-stack/01-ai-coding/agent-skill-pairing.md` (status `research`) — new tier-2 pattern doc. Contents: the pattern in one sentence, why it works (links back to proactive-patterns reliability section), table of existing workspace pairings as canonical demos, ideal pairings for development work (pre-design crawler / code review / planning / scaffolding / curation / docs / test triage), anti-patterns, decision flow.
- Cross-references the proactive-patterns Firing Reliability section, the skill-patterns Purpose caveat, and AGENTS.md §"Subagent Limitation" so the three docs together form a coherent design-pattern cluster.

**Why it mattered:** the firing-reliability principle was a *negative* finding ("don't put triggers in skills"). Agent + skill pairing is the *positive* counterpart ("here's how to actually deliver skill content reliably"). Documenting only the negative leaves a gap; pairing this with the positive pattern makes the guidance complete.

**Tier decision:** tier-2-clean. This is general Claude Code design wisdom, not cybersader-specific infrastructure. Ships to the agentic public mirror per the existing 02-stack/ allowlist.
