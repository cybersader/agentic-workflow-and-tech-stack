## Session 5: Obsidian handoff — obs-open / obs-uri

Obsidian is a core system across this stack; the missing piece was the **agent→human handoff**: when the agent creates or references a doc, the human shouldn't vault-spelunk to find it. Shipped `obs-open <path>` (opens the doc in the Obsidian GUI from WSL) and `obs-uri <path>` (prints a clickable `obsidian://` link), appended to `local-search-helpers.sh`.

Mechanics discovered by testing (not assumed): the Obsidian CLI's `open` command needs a registered-vault *name*, which doesn't reliably map from a filesystem path (`vault=mcp-workflow-and-tech-stack` → "Vault not found" despite the vault being real) — so the helpers use the **absolute-path URI form** (`obsidian://open?path=<percent-encoded Windows path>`), which makes Obsidian resolve the owning vault itself. `wslpath -w` converts, `powershell.exe Start-Process` launches (`cmd /c start` breaks on `&` in URIs — verified failure). Both helpers tested live end-to-end.

Wiring per the firing-reliability doctrine: **CLAUDE.md carries the behavior steer** ("hand docs to the human via obs-open/obs-uri, never bare paths" — one line, must-fire tier), the `local-search` skill carries the how, and the tier-2 pattern doc got an "Obsidian handoff" section (extended the existing doc — supersession rule, no new file).

Follow-up candidates (not built): `obsidian://search?query=` handoff for "look at these results in your vault"; Advanced-URI plugin integration for heading/line-level deep links if plain file-open proves too coarse.
