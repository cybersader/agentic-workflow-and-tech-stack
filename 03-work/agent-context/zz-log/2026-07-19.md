## Session: two WSL freezes — sampler identified both culprits

The existing `~/wsl-sampler.sh` (3-second `/proc` snapshots mirrored to Windows each cycle) finally paid off. WSL appeared to "hit 95% CPU and become untouchable" twice. Live state after restart was healthy (23 GiB total, 21–22 GiB available, no swap, no containers), but historical samples were decisive:

### Incident A — direct ck MCP bypassed the shell cap

A Claude session called ck-search `hybrid_search` against a small (~150 MB) application repo. The MCP server is configured globally as direct `ck --serve`, so it bypasses `_cks_with_cap` in `local-search-helpers.sh` (the documented 8 GiB `ulimit -v` only wraps shell helpers).

- ck RSS: **17.1 GB / 69.6%**
- load average: **117**
- swap: **fully exhausted**
- memory PSI `some`/`full`: **~99%**
- kernel OOM counter: 0 in this episode — the machine thrashed rather than killing ck

The full-day sampler showed this was not isolated: ck also reached ~16, ~7.5, ~12, and ~10 GiB in earlier episodes. The index itself was only ~13 MB — the blow-up is runtime/indexing behavior, not corpus size.

**Response:** user explicitly chose to ditch ck for now. Removed the global `ck-search` MCP entry via `claude mcp remove`, killed the live server, preserved the exact server config at `~/.claude/disabled-mcp/ck-search-2026-07-19.json` plus a full pre-change `~/.claude.json` backup. Existing `.ck/` indexes were NOT deleted. `CLAUDE.md` search routing now quarantines `ck`, MCP, and all `cks*` helpers until the replacement IR project ships or a process-level memory guard is verified.

### Incident B — unbounded planning-agent filesystem scans

A separate new-project planning session launched three read-only Explore agents. One agent searched for reference code by recursively scanning all of the Linux home directory, the entire Windows Documents tree, and later Claude transcript/cache trees. This was far outside the named repo scope. Two Claude worker processes (process title was the installed Claude Code version) grew to roughly **13.2 GiB + 5.4 GiB**, exhausted swap, held memory PSI at ~99%, and incremented the kernel OOM counter.

The product's Rust code was not running and was not the cause. The failure was agent search breadth / output buffering. The applicable rule: **never run recursive `find`/`grep -R` over all of `~/`, Windows `Documents`, or `~/.claude`; scope to a named repo or bounded path, and use dedicated read/search tools.** Agent count caps do not prevent one agent from exhausting memory.

## Current state

## Session (cont.): context-overflow rails — pconv handoff + Sol compaction + launcher recovery

Two more overflow strandings today (kwir mid-`Skill(update-config)` injection; speakr after a 1h+ verification turn) proved the pattern: near 200k on Sol, one large mid-turn tool result crosses the hard limit, and `/compact` then 400s because compaction submits the oversized conversation. Docs-verified (claude-code-guide agent): trigger ≈ `autoCompactWindow − 20k output reserve − 13k`, no official local compaction exists. Shipped three rails:

1. **`pconv handoff`** (portaconv `2f14515`): deterministic, local, byte-bounded recovery docs. Pins the first substantive user task; keeps newest interaction units; omits thinking; abbreviates injected Skill bodies (structural detection: preceding `Skill` tool_use + markers — a plain user-pasted doc is NOT eaten) and system-reminder envelopes (embedded mid-turn user messages extracted); head/tail truncation at UTF-8 boundaries (results tail-biased so error lines survive); exact ≤48 KiB enforcement on rendered bytes. 8 unit tests incl. determinism, emoji-boundary, recency-bias; validated on both real overflow sessions (kwir 155 KB tail → 10.5 KB; speakr 1286 msgs → 49,088 B). Never touches the source JSONL.
2. **Sol-only earlier compaction**: `profiles/claude-global/sol-settings.json` (`autoCompactWindow: 150000` → trigger ~117k, ~50k emergency headroom) injected by `claudex()` only — caller `--settings` overrides win; fablex (Fable 1M main) and native cc untouched. Snippet dir resolved at source time (BASH_SOURCE is cwd-relative — call-time resolution broke after any `cd`; caught in stub tests).
3. **Recovery UX**: `cc-handoff`/`cchd` (dry-run `-n`, explicit-ID or newest-for-cwd, launcher choice, fails closed — no blank session on generation failure) + `x`/`ccx` session option **"Recover overflowed session"** (pconv picker → bounded handoff → fresh launch with a pointer prompt; requires `pconv handoff` capability, no silent fallback to `--tail`).

Verified: `cargo test` 37+18 green, clippy clean, `bash -n` both snippets, stubbed launch tests (injection present by default / absent with explicit --settings / absent in fablex; ccx handoff path generates-then-launches-fresh with perms+tier intact).

## Session (cont.): launcher memory — "same as last time" + cchd brain reuse

Owner UX ask: after an overflow, "exit → x → get back to working" should reuse the previous launch config automatically. Added per-directory launcher state (`~/.claude/launcher-state/<encoded-cwd>.last`, one line: `brain= tier= perms=`), written on every real ccx launch. `x` now offers **"Same as last time"** as the Enter default when state exists (only the session question is asked); `cchd` defaults its launcher/tier/perms to the recorded state (explicit launcher arg still wins, claudex fallback when no state). Verified via stubbed launches: state records, cchd picks it up (`brain=cc tier=opus perms=skip` → `CLAUDE_CODE_SUBAGENT_MODEL=opus claude --dangerously-skip-permissions`), ccx "last" fast path skips the perms/tier menus.

### Fix: cchd dropped skip-perms when no launcher state existed

Field report (dataflowy): session originally launched in bypass mode but NOT via ccx → no launcher-state file → cchd fell back to bare `claudex` with no perms flag → recovered session came up in manual/ask mode. Fix: missing/absent `perms` now defaults to `skip` (the house y-variant default); only a *recorded* `perms=ask` produces prompting mode. Verified both branches with stubs.

### Perf: handoff generation was silently slow (2m43s → ~12-33s) + progress lines

Field report (cyberchaste): recovery looked frozen. Cause: `pconv` `load()` content-scanned every JSONL under `~/.claude/projects/` to locate a session — minutes on DrvFS. Fast path added: the home file (`<id>.jsonl`) is findable by directory listing alone and always wins pick_rank, so the content scan now runs only for the no-home-file compact-continuation edge case. Cyberchaste: 2m43s → 12s (explicit uuid, the ccx path) / 33s (`--latest`, dominated by the resolution metadata scan). Also: stderr progress from pconv (resolving → loading N messages → rendering) and a "generating bounded handoff…" line in cchd/ccx so the wait is visibly working. Remaining ~10-30s is DrvFS metadata reality — the same measurement that motivates moving hot trees WSL-native.
